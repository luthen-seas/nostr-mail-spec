# NIP-24: Extra Metadata Fields and Tags

## Status
Active (draft, optional)

## Summary
NIP-24 documents optional metadata fields and tags that have become de facto standards in the Nostr ecosystem but are not formally specified in other NIPs. It covers extra fields for `kind:0` user metadata events (display_name, website, banner, bot, birthday), deprecated fields, deprecated `kind:3` relay list data, and universally applicable tags (r, i, title, t).

## Motivation
As the Nostr ecosystem grew, clients began using additional metadata fields and tags beyond what NIP-01 formally defined. NIP-24 standardizes these conventions to ensure interoperability. Without this NIP, each client would invent its own field names for common needs like display names, website URLs, banner images, and hashtags, leading to fragmentation.

## Specification

### Event Kinds

NIP-24 does not define new event kinds. It extends the following existing kinds:

| Kind | Extended With |
|------|--------------|
| 0 | Additional metadata fields in content JSON |
| 3 | Deprecated relay list format (use NIP-65 instead) |
| Any | Universal tags (r, i, title, t) |

### Kind 0 (User Metadata) Extra Fields

These fields are added to the stringified JSON object in the `content` of `kind:0` events:

| Field | Type | Description |
|-------|------|-------------|
| `display_name` | string | An alternative, bigger name with richer characters than `name`. The `name` field should always be present as a fallback. |
| `website` | string | A URL related to the event author |
| `banner` | string | A wide image URL (approximately 1024x768) for optional profile background display |
| `bot` | boolean | `true` if the account's content is generated by automation (bot) |
| `birthday` | object | Object with optional `year`, `month`, and `day` fields |

**Deprecated fields (do not use in new implementations):**
| Deprecated Field | Replacement |
|-----------------|-------------|
| `displayName` | `display_name` |
| `username` | `name` |

### Tags

#### `r` tag (web URL reference)
```json
["r", "<URL>"]
```
References a web URL that the event is about or related to. Can be used on any event kind.

#### `i` tag (external identifier)
```json
["i", "<external-identifier>"]
```
References an external identifier per NIP-73. Can be used on any event kind.

#### `title` tag
```json
["title", "<name>"]
```
A human-readable name/title. Used by:
- NIP-51 sets/lists
- NIP-52 calendar events
- NIP-53 live events
- NIP-99 classified listings

#### `t` tag (hashtag)
```json
["t", "<lowercase-hashtag>"]
```
A hashtag. The value MUST be lowercase. Can be used on any event kind.

### Kind 3 Deprecated Relay Format

The `content` field of `kind:3` events previously contained a relay list:
```json
{
  "wss://relay1.example.com": {"read": true, "write": true},
  "wss://relay2.example.com": {"read": true, "write": false}
}
```

**This format is deprecated.** Use NIP-65 (`kind:10002`) for relay list metadata instead.

## Implementation Notes

1. **`name` vs `display_name`:** The `name` field (from NIP-01) should always be present as a simple, short identifier (like a username). `display_name` is optional and can contain richer characters, spaces, and emoji. Clients should fall back to `name` if `display_name` is absent.

2. **Hashtag normalization:** The `t` tag value MUST be lowercase. Clients should normalize hashtags before creating events (e.g., `#Nostr` in content becomes `["t", "nostr"]` in tags).

3. **Banner image dimensions:** The banner image should be approximately 1024x768 pixels (wide format). Clients may display it as a profile header/cover image.

4. **Bot flag:** The `bot` field helps clients identify automated accounts. Clients may display a bot indicator badge or filter bot content.

5. **Birthday privacy:** The `birthday` object has all fields optional. Users can share just year, just month+day, or the full date.

6. **Deprecated fields:** Clients reading `kind:0` events should check for both `display_name` and the deprecated `displayName`, preferring `display_name`. Similarly, check both `name` and the deprecated `username`.

## Client Behavior

- Clients SHOULD display `display_name` when available, falling back to `name`
- Clients SHOULD display banner images in profile views when available
- Clients SHOULD indicate bot accounts with a visual badge when `bot` is `true`
- Clients SHOULD normalize hashtags to lowercase before adding `t` tags
- Clients SHOULD use NIP-65 for relay lists (not the deprecated kind:3 content format)
- Clients MAY display birthday information in profiles
- Clients MAY use `r` tags for link previews or reference tracking

## Relay Behavior

- Relays MUST index `t` tags for hashtag queries (`#t` filter)
- Relays MUST index `r` tags for URL queries (`#r` filter)
- Relays MUST index `i` tags for external identifier queries (`#i` filter)
- Relays SHOULD support kind:0 as a replaceable event per NIP-01
