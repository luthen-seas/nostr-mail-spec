## Tier 2: Should Do for Credibility (High Priority)

### H2.1: NIP-44 Compatibility Verification

Our encryption must be byte-compatible with the canonical NIP-44 implementation.

| Task | Action | Acceptance Criteria |
|------|--------|-------------------|
| [ ] H2.1.1 | Download NIP-44 official test vectors from nostr-protocol/nips | Vectors obtained |
| [ ] H2.1.2 | Run nostr-tools nip44 encrypt/decrypt against official vectors | All pass |
| [ ] H2.1.3 | Run go-nostr nip44 encrypt/decrypt against official vectors | All pass |
| [ ] H2.1.4 | TS encrypts, Go decrypts (same conversation key) | Round-trip succeeds |
| [ ] H2.1.5 | Go encrypts, TS decrypts (same conversation key) | Round-trip succeeds |

**Why this matters**: If our NIP-44 usage doesn't match the canonical implementation, no existing NOSTR client will be able to interact with our messages. This is the foundation everything else stands on.

### H2.2: Cross-Implementation Round-Trip (Live)

Actual serialized events passing between TypeScript and Go.

| Task | Action | Acceptance Criteria |
|------|--------|-------------------|
| [ ] H2.2.1 | TS creates kind 1400 rumor → serialize to JSON → Go parses | Fields match |
| [ ] H2.2.2 | Go creates kind 1400 rumor → serialize to JSON → TS parses | Fields match |
| [ ] H2.2.3 | TS wraps message → serialize kind 1059 to JSON → Go unwraps | Content, sender, verification all match |
| [ ] H2.2.4 | Go wraps message → serialize kind 1059 to JSON → TS unwraps | Content, sender, verification all match |
| [ ] H2.2.5 | TS state event → serialize tags → Go deserializes → merge | States match after round-trip |
| [ ] H2.2.6 | Go state event → serialize tags → TS deserializes → merge | States match after round-trip |

**Why this matters**: This is the proof that two independent implementations can interoperate. Without it, the NIP is untested theory.
